Harness Week — Prototype Demo

My Interpretation of Harness

Specs as feedforward. Gates as feedback. A small Greenfield CRUD, built to prove the loop end-to-end — not to ship a product.

martín — harness-v1
01 — What Harness Means To Me

The automated control system around a coding agent.

  • Specs guide it before it writes code — feedforward.
  • Gates verify after what it produced — feedback.
  • You don't review every line. You design the conditions.

02 — Why Minimalist

One small Greenfield case, not a big one half-done.

  • No multi-repo, no Brownfield, no integrations — on purpose.
  • This week's goal is to validate the concept, not ship a product.
  • A complete cycle on something small beats a partial cycle on something big.

The app

03 — Spec-Driven Development

The spec is the primary artifact. The code is its expression.

constitution.md
Global rules — layering, naming, standard error handling. Non-negotiable per feature.
products-spec.md
The feature itself. The WHAT, not the HOW — no class names, no ORM.
verification.md
Checkable criteria, one ID each, traced 1:1 to a test in /gates.
03 — What A Gate Actually Is

A tollgate. Doesn't matter who wrote the code.

Functional gates

13

Black-box HTTP tests, one per acceptance criterion. Zero compile-time reference to /src — the gate builds before the app exists.

Architecture gates

5

NetArchTest reads the compiled assembly, not the source — real dependency direction between layers, not just syntax.
